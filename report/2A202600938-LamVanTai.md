## 1. Tổng Quan Hệ Thống

Bài lab xây dựng một **Shopping Assistant** theo mô hình multi-agent dùng **LangGraph**, phục vụ cho nền tảng thương mại điện tử giả lập VinShop Demo.

Hệ thống có khả năng:
- Trả lời câu hỏi về **chính sách** (hoàn trả, giao hàng, voucher) qua RAG
- Tra cứu **dữ liệu thực** (đơn hàng, khách hàng, voucher) qua lookup tools
- Kết hợp cả hai để trả lời câu hỏi phức hợp
- Xử lý trường hợp thiếu thông tin (`clarification_needed`) và không tìm thấy (`not_found`)

## 2. Kiến Trúc Multi-Agent

```
User
 │
 ▼
Supervisor Agent
 │  (phân tích câu hỏi, quyết định route)
 ├──────────────────────┬──────────────────────┐
 ▼                      ▼                      │
Worker 1               Worker 2               │
Policy/RAG Agent       Data Lookup Agent       │
 │  (search Chroma)     │  (4 lookup tools)    │
 └──────────┬───────────┘                      │
            ▼                                  │
       Worker 3 ◄─────────────────────────────┘
       Response Agent
       (tổng hợp final answer)
            │
            ▼
       Final Answer
```

**Luồng xử lý:**
1. User gửi câu hỏi
2. Supervisor phân tích → trả JSON `{needs_policy, needs_data, status}`
3. LangGraph route đến worker phù hợp (policy only / data only / both)
4. Workers xử lý và ghi kết quả vào shared state
5. Response Worker tổng hợp câu trả lời cuối cùng có Evidence

## 3. Các File Đã Hoàn Thiện

### 3.1. `src/rag/parser.py` — Chunking Policy Markdown

Parse file `policy_mock_vi.md` theo cấu trúc heading:
- Nhận diện `## Heading 2` và `### Heading 3`
- Mỗi chunk giữ: `section_h2`, `section_h3`, `citation`, `rendered_text`
- Kết quả: **38 chunks** từ 10 section chính của policy

```python
# Ví dụ chunk được tạo ra:
{
    "section_h2": "5. Chính sách đổi trả và hoàn tiền",
    "section_h3": "5.1. Điều kiện chung để gửi yêu cầu",
    "citation": "policy_mock_vi.md > 5. Chính sách đổi trả và hoàn tiền > 5.1. Điều kiện chung để gửi yêu cầu",
    "rendered_text": "5. Chính sách đổi trả...\n5.1. Điều kiện chung...\nKhách hàng có thể gửi..."
}
```

### 3.2. `src/rag/vector_store.py` — ChromaDB Vector Store

- Dùng `SentenceTransformerEmbeddings` với model `all-MiniLM-L6-v2`
- Lưu vector vào `ChromaPersistentClient` tại `src/.chroma`
- `ensure_index()`: tự động build nếu collection rỗng
- `search()`: embed query → cosine similarity → trả top-k hits có citation và distance

### 3.3. `src/app/data_access.py` — Lookup Tools

Load JSON một lần, build 4 index để lookup O(1):

| Tool | Mô tả |
|---|---|
| `get_customer_by_id(customer_id)` | Thông tin khách hàng, hạng, quota voucher |
| `get_orders_by_customer_id(customer_id)` | Danh sách đơn hàng gần nhất |
| `get_order_detail_by_order_id(order_id)` | Chi tiết đơn: trạng thái, hạn trả hàng, `can_return_now` |
| `get_vouchers_by_customer_id(customer_id, only_active)` | Danh sách voucher, filter active |

Mỗi tool trả `{"status": "ok", ...}` hoặc `{"status": "not_found", ...}`.

### 3.4. `src/app/prompts.py` — Prompts Tách Riêng

4 prompts riêng biệt cho từng agent, tối ưu token:
- `SUPERVISOR_PROMPT`: yêu cầu trả JSON nhỏ để parse dễ
- `POLICY_WORKER_PROMPT`: bắt buộc gọi `search_policy` tool trước
- `DATA_WORKER_PROMPT`: hướng dẫn dùng đúng tool theo thông tin có trong câu hỏi
- `RESPONSE_WORKER_PROMPT`: 3 format output cố định (Success / Clarification / Not found)

### 3.5. `src/app/graph.py` — LangGraph Orchestration

**StateGraph** với shared state `ShoppingState`:

```python
class ShoppingState(TypedDict, total=False):
    question: str
    route: dict        # output của supervisor
    policy_result: dict  # output của worker 1
    data_result: dict    # output của worker 2
    final_answer: str
    trace: Annotated[list, operator.add]  # append-only trace
```

**Conditional routing:**
```
supervisor
  ├─[needs_policy only]──→ worker_1_policy ──→ worker_3_response
  ├─[needs_data only]────→ worker_2_data ────→ worker_3_response
  ├─[both]───────────────→ worker_1_policy → worker_2_data → worker_3_response
  └─[clarification]──────→ worker_3_response
```

**Agentic loop trong mỗi worker:**
- Worker bind tools với LLM
- LLM tự quyết định gọi tool nào, bao nhiêu lần
- Tool output được feed lại vào messages
- Tối đa 3-5 vòng để tránh infinite loop

## 4. Kết Quả Chạy Thử

### 4.1. Câu hỏi Policy
```
Q: Chính sách hoàn trả hàng ra sao?

Answer: Khách hàng có thể gửi yêu cầu trả hàng trong vòng 15 ngày
kể từ khi đơn cập nhật giao hàng thành công...

Evidence:
- Policy: policy_mock_vi.md > 5. Chính sách đổi trả > 5.1. Điều kiện chung
```

### 4.2. Câu hỏi Data
```
Q: Đơn hàng 1971 bao giờ được giao?

Answer: Đơn hàng 1971 đang trong quá trình giao hàng (in_transit),
dự kiến giao ngày 09/06/2026 qua SPX Express...

Evidence:
- Order data: order_status=in_transit, estimated_delivery=2026-06-09
```

### 4.3. Câu hỏi Kết Hợp
```
Q: Đơn hàng 1971 có được hoàn trả không?

Answer: Đơn hàng 1971 hiện đang ở trạng thái in_transit (chưa giao thành công),
do đó chưa đủ điều kiện để bắt đầu quy trình trả hàng theo chính sách 5.10.

Evidence:
- Policy: policy_mock_vi.md > 5.10. Quan hệ giữa trạng thái đơn và quyền trả hàng
- Order data: can_return_now=false, order_status=in_transit
```

### 4.4. Clarification Needed
```
Q: Voucher của tôi còn dùng được không?

Status: clarification_needed
Question: Anh/chị vui lòng cung cấp mã khách hàng (customer_id)
để em kiểm tra thông tin voucher chính xác.
```

### 4.5. Not Found
```
Q: Kiểm tra đơn hàng 9999 giúp tôi

Status: not_found
Message: Không tìm thấy đơn hàng với ID '9999' trong hệ thống.
```
